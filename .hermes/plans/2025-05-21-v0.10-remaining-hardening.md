## Task 1: Fix crucible.rs — chars.next().unwrap() panics (5 occurrences)

**Objective:** The markdown parser calls `.next().unwrap()` on character iterators inside `while let Some(c) = chars.next()` blocks, but the `.next()` calls inside the loop body can return `None` if parsing reaches end of string mid-pattern.

**Files:**
- Modify: `src/crucible.rs:805, 821, 838, 850, 860`

**Step 1: Read the file around the affected lines**

Run: `cat -n src/crucible.rs | sed -n '795,870p'`

**Step 2: Replace each `chars.next().unwrap()` with `match chars.next() { Some(c) => c, None => break }`**

At lines ~805, 821, 838 — these are inside character collection loops like:
```rust
// BEFORE (panics if string ends mid-pattern):
inner.push(chars.next().unwrap());

// AFTER (graceful break):
match chars.next() {
    Some(c) => inner.push(c),
    None => break,
}
```

At lines ~850, 860 — same pattern for `link_text` and `url`:
```rust
// BEFORE:
link_text.push(chars.next().unwrap());
url.push(chars.next().unwrap());

// AFTER:
match chars.next() {
    Some(c) => link_text.push(c),
    None => break,
}
// and
match chars.next() {
    Some(c) => url.push(c),
    None => break,
}
```

**Step 3: Verify compilation**

Run: `cargo check`
Expected: 0 errors, 0 warnings

**Step 4: Commit**

```bash
git add src/crucible.rs
git commit -m "fix(crucible): replace chars.next().unwrap() with match-based safe iteration"
```

**Step 2: Replace production code to_str().expect() with proper error handling**
